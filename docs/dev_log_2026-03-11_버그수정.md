## 1. PIR 센서 이벤트 폭주 해결

### 문제
- `esp32_home1`의 PIR 센서가 **쿨다운 없이** 매 rising edge마다 TCP 이벤트를 전송
- 초당 수십 건의 `motion_detected` 이벤트가 서버로 유입 → 서버 TCP 버퍼 과부하
- 웹 UI에 알림 팝업이 끊임없이 표시되어 다른 조작 불가

### 원인
- `esp32_home1_hmac.ino`의 `handlePir()` 함수에 쿨다운 로직 미적용
- `esp32_home2`에는 `PIR_ALERT_COOLDOWN_MS` (30초) 쿨다운이 있었으나 `home1`에는 누락
- HC-SR501 PIR 센서 신호 노이즈로 HIGH/LOW가 빠르게 토글 → rising edge 다수 발생

### 수정 내용

#### 파일: `esp32/esp32_home1_hmac/esp32_home1_hmac.ino`
- **30초 쿨다운 추가**: `PIR_ALERT_COOLDOWN_MS` 이내 중복 전송 차단
- **500ms 디바운싱 추가**: PIR 신호가 500ms 이상 안정되어야 상태 전환으로 인정

```cpp
// 디바운싱: 500ms 이상 안정된 신호만 유효 처리
static bool stableMotion = false;
static bool lastRaw      = false;
static unsigned long lastChangeTime = 0;

if (rawMotion != lastRaw) {
    lastChangeTime = now;
    lastRaw = rawMotion;
}
if ((now - lastChangeTime) < 500) return;  // 안정화 대기
```

#### 파일: `server/tcp_server.py`
- 일반 `motion_detected` 이벤트는 서버 로그만 남기고 **웹 알림 팝업을 차단**
- `guard_alert` (방범 침입) / `presence_alert` (장시간 정적)만 팝업 표시

```python
# 일반 motion_detected 는 로그만 남기고 웹 알림을 보내지 않음
if event == "motion_detected":
    return
```

## 2. TCP 재접속 시 디바이스 등록 해제 버그 수정

### 문제
- ESP32가 재접속할 때 `/devices` API에서 장치 수가 0으로 표시
- 웹 UI에서 명령 전송은 성공으로 표시되나 실제 센서 동작 안 됨

### 원인
- `_on_register()`에서 `old.close()`로 이전 연결을 닫으면 해당 핸들러의 `finally` 블록에서 `_on_disconnect()` 호출
- `_on_disconnect()`가 `_registry.pop(device_id)`를 실행하여 **이미 등록된 새 클라이언트까지 제거**

### 수정 내용

#### 파일: `server/tcp_server.py`

```python
async def _on_disconnect(self, client, addr):
    if client:
        # 재접속으로 새 클라이언트가 이미 등록된 경우,
        # 이전 연결 해제 시 새 클라이언트를 제거하지 않음
        current = self._registry.get(client.device_id)
        if current is client:
            self._registry.pop(client.device_id, None)
            self._unified_state_manager.remove(client.device_id)
        client.close()
```

## 3. STT 엔진 LISTENING 상태 멈춤 수정

### 문제
- 마이크 버튼을 몇 번 누르면 서버가 응답하지 않음
- 웹페이지 새로고침도 안 되는 완전 멈춤 상태

### 원인
1. `activate()` (버튼 클릭)이 `_state`만 `LISTENING`으로 변경하지만, `listen_start`가 `_process_loop` 내부의 **로컬 변수**라 설정 불가 → 타임아웃이 작동하지 않아 LISTENING에 영구 stuck
2. Whisper 추론이 동시에 여러 건 실행되어 CPU/메모리 과부하 → asyncio 이벤트 루프 블로킹 → uvicorn HTTP 응답 불가

### 수정 내용

#### 파일: `server/stt_engine.py`

**① `listen_start` 인스턴스 변수화**
- `_process_loop`의 로컬 변수 `listen_start`를 `self._listen_start` 인스턴스 변수로 변경
- `activate()`에서도 `self._listen_start = time.time()` 설정 가능

**② Whisper 동시 실행 방지 + 타임아웃**

```python
async def _transcribe(self, audio, speech_start=None):
    if self._transcribing:
        logger.warning("[STT] Whisper 추론 중 — 중복 요청 무시")
        return
    self._transcribing = True
    try:
        await self._do_transcribe(audio, speech_start)
    finally:
        self._transcribing = False  # 어떤 경로로든 반드시 해제
```

**③ Whisper 30초 타임아웃**

```python
result = await asyncio.wait_for(
    loop.run_in_executor(None, self._run_whisper_with_debug, audio),
    timeout=30.0
)
```

**④ 명령 처리 중 웨이크워드/버튼 입력 차단**

Whisper 추론 + LLM 파싱 + ESP32 명령 전송(`_transcribing == True`) 중에 새 웨이크워드나 버튼 입력이 들어오면 LISTENING 전환을 차단하여 큐 폭주를 방지한다.

```python
# activate() — 버튼 클릭 시
def activate(self):
    if self._transcribing:
        logger.info("[STT] 명령 처리 중 — 버튼 무시")
        return
    if self._state == STATE_IDLE:
        ...

# _process_loop() — Porcupine 웨이크워드 감지 시
if self._detect_porcupine(pcm_int16) and not self._transcribing:
    self._state = STATE_LISTENING
    ...
```

## 4. LLM 파싱 타임아웃 및 실패 시 TTS 안내 추가

### 문제
- 음성 명령 처리 시 LLM 응답이 느릴 때 사용자가 무한 대기
- 파싱 실패 시 에러 메시지만 반환되고 TTS 음성 안내 없음

### 수정 내용

#### 파일: `server/command_router.py`

- LLM 파싱에 **8초 타임아웃** 적용
- 타임아웃 시: TTS `"처리 시간이 너무 오래 걸렸어요. 다시 말씀해주시겠어요?"`
- 파싱 실패 시: TTS `"잘 알아듣지 못했어요. 다시 말씀해주시겠어요?"`

```python
try:
    data = await asyncio.wait_for(self._llm.parse(text), timeout=8.0)
except asyncio.TimeoutError:
    return {"status": "timeout", "tts_response": "처리 시간이 너무 오래 걸렸어요. 다시 말씀해주시겠어요?"}

if not data:
    return {"status": "unknown", "tts_response": "잘 알아듣지 못했어요. 다시 말씀해주시겠어요?"}
```

## 5. kill_server.sh / run_server.sh 단계적 종료 보강

### 문제
- `kill_server.sh` 실행 후 바로 `run_server.sh` 실행 시 포트 충돌 (`Address already in use`)
- `run_server.sh`의 기존 프로세스 정리가 단순 `kill` + `sleep 1`이라 포트 해제 전에 새 서버가 시작되는 경우 발생

### 수정 내용

#### 파일: `kill_server.sh`

- `kill_port()` 함수 도입: **SIGTERM → 최대 5초 대기 → SIGKILL** 단계적 종료
- 각 포트별 프로세스가 실제로 종료될 때까지 확인 후 다음 진행

#### 파일: `run_server.sh`

- `kill_server.sh`와 동일한 `kill_port()` 함수를 추가하여 시작 전 기존 프로세스 정리 로직 통일
- `set -e` 환경에서 `lsof`/`kill` 실패 시 스크립트 종료되는 문제 방지 (`|| true`)

```bash
kill_port() {
    local port=$1
    local label=$2
    local pids
    pids=$(lsof -t -i:"$port" 2>/dev/null || true)
    if [ -z "$pids" ]; then return; fi
    kill $pids 2>/dev/null || true
    for _ in $(seq 1 10); do
        if ! lsof -t -i:"$port" &>/dev/null; then
            echo "  $label 종료 완료 (port $port)"; return
        fi
        sleep 0.5
    done
    pids=$(lsof -t -i:"$port" 2>/dev/null || true)
    if [ -n "$pids" ]; then
        kill -9 $pids 2>/dev/null || true
    fi
}
```
